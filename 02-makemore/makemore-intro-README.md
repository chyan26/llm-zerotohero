# The Spelled-out Intro to Language Modeling: Building makemore

> 筆記整理自 Andrej Karpathy《Neural Networks: Zero to Hero》系列 — 第一集
> 影片連結：[YouTube](https://www.youtube.com/watch?v=PaCmpygFfXo&list=PLAqhIrjkxbuWI23v9cThsA9GvCAUhRvKZ)

## 一、專案目標

`makemore` 是一個字元級（character-level）語言模型，用來「生成更多」（make more）像訓練資料一樣的字串。本集使用的資料集是 `names.txt`（美式人名列表），目標是訓練一個模型能生成聽起來像人名的新字串。

這一集只實作最簡單的版本：**Bigram 語言模型**——只看「前一個字元」來預測「下一個字元」。

## 二、兩種實作路徑

### 1️⃣ 計數法（Counting-based）

* 建立一個 27×27 的計數矩陣 `N`（26 個字母 + 特殊字元 `.` 代表開始/結束）
* 遍歷所有人名，統計每一對相鄰字元 (bigram) 出現的次數
* 對每一列做 normalize（除以總和）得到機率分布 `P`
* 使用 `torch.multinomial` 依機率取樣，逐字元生成新名字
* 用負對數概似（Negative Log-Likelihood, NLL）評估模型好壞：
* 機率越高 → log 值越接近 0 → loss 越低
* 對資料集中不存在的 bigram，機率為 0 會導致 loss 變成無限大 → 需要 **加一平滑（Laplace/add-one smoothing）**



### 2️⃣ 神經網路法（Gradient-based）

* 將每個字元用 **one-hot 編碼** 表示為輸入向量
* 建一個 27×27 的權重矩陣 `W`（隨機初始化）
* 前向傳播：
1. `logits = one_hot(x) @ W`
2. `counts = logits.exp()`（模擬計數，永遠是正數）
3. `probs = counts / counts.sum(1, keepdims=True)`（normalize，即 **softmax**）


* 用 **cross entropy / NLL** 作為 loss function
* 用梯度下降（gradient descent）反覆更新 `W`：
* `loss.backward()` 算出梯度
* `W.data += -lr * W.grad`


* 加入 **正則化項（regularization）**：在 loss 中加入 `W` 的平方和，讓權重不會過大，效果等同於計數法中的平滑

## 三、關鍵概念

| 概念 | 說明 |
| --- | --- |
| **Bigram** | 只用前一個字元預測下一個字元的最簡單語言模型 |
| **One-hot encoding** | 把字元轉成稀疏向量，方便丟進神經網路 |
| **Softmax** | 把任意實數向量轉成合法機率分布（exp 再 normalize） |
| **Negative Log-Likelihood** | 衡量模型對真實資料預測好壞的損失函數，也是 MLE 的目標函數 |
| **Gradient Descent** | 透過反覆計算梯度、更新參數，逐步逼近最小 loss 的解 |
| **Smoothing / Regularization** | 避免機率為 0 或權重過大，兩種方法在數學上等價 |

## 四、核心結論

計數法與神經網路法本質上是在解**同一個最佳化問題**：

* 計數法：直接用**解析解**算出最大概似估計（MLE）
* 神經網路法：用**梯度下降**迭代逼近同一個 MLE 解

差別在於：計數法只適用於這種「小型、離散、可窮舉」的 bigram 情境；神經網路法的架構（one-hot → 矩陣乘法 → softmax → NLL）則可以自然擴展到更長的 context、更深的網路、更複雜的非線性關係——這正是後續影片（MLP、RNN、Transformer）要解決的問題。

## 五、學員核心提問與 Debug 實錄 (新增段落)

### ❓ 提問 1：統計 Bigram 的核心程式碼到底在做什麼？

```python
stoi['.'] = 0
for w in words:
  chs = ['.'] + list(w) + ['.']
  for ch1, ch2 in zip(chs, chs[1:]):
    ix1 = stoi[ch1]
    ix2 = stoi[ch2]
    N[ix1, ix2] += 1

```

* **原理解析**：這是統計字母兩兩相鄰配對（Bigram）次數的過程。例如單字 `w = "cat"`，`chs` 會被前後加上特殊符號變為 `['.', 'c', 'a', 't', '.']`。透過 `zip(chs, chs[1:])` 進行兩兩相鄰分組（`('.', 'c')`, `('c', 'a')`...），將字元透過字典 `stoi` 轉成數字索引後，在二維矩陣 `N[ix1, ix2]` 的對應格子裡做加總記數。

### ❓ 提問 2：既然有用 Generator 控制 Seed，為什麼還要叫 Random？

```python
g = torch.Generator().manual_seed(2147483647)
p = torch.rand(3, generator=g)

```

* **原理解析**：電腦中的隨機為「偽隨機」，是由複雜公式計算出來的。`torch.rand` 的職責是提供**亂數的分佈特性**（製造均勻的混亂）；而 `torch.Generator().manual_seed(...)` 是為了**控制混亂的起點**（固定隨機數密碼表的指針）。在機器學習開發與 Debug 中，這樣做能確保「實驗可重複性」、讓 Bug 能穩定重現，並且隔離局部代碼，避免全域隨機性互相干擾。

### ❓ 提問 3：為什麼實作多項式抽樣時，我抽到的字元跟範例影片對不上？

```python
ix = torch.multinomial(p, num_samples=1, replacement=True, generator=g).item()

```

* **原理解析**：
1. **版本差異**：PyTorch 2.x 版本更新了 `torch.multinomial` 的底層離散格點採樣算法，即使 Seed 相同，2.x 抽出的數值與舊版 1.x 就是會不一致。
2. **硬體與多項式特性**：CPU 與 Mac GPU (MPS) 的底層隨機生成算法是獨立的。此外，`multinomial` 抽樣就像轉輪盤，雖然預測 `m` 的機率最高，但機率低的字元（如 `c`）並不代表中獎率是 0%。只要大數法則下抽樣次數（如 10,000 次）放大，實際抽中比例就會完美收斂回原本的機率分佈。



### ❓ 提問 4：一次性計算整張機率表 `P /= P.sum(1, keepdims=True)` 的意義何在？跟之前單行除法差在哪？

* **原理解析**：
* **舊做法**：`p = N[0] / torch.sum(N[0])` 屬於「臨時抱佛腳」，每當模型遇到新字元，才即時呼叫顯示卡或 CPU 做一次除法，在迴圈中效率極低。
* **新做法**：一次性把整個 $27 \times 27$ 的矩陣交給最擅長平行運算的晶片，利用**廣播機制**將每一行同時進行機率標準化，開好一張永久的「機率建檔查找表 `P`」，後續抽樣時只需進行 $O(1)$ 的極速查找。



### ❓ 提問 5：為什麼標準化矩陣時必須加 `keepdim=True`？不加的後果是什麼？

* **原理解析**：與 Python/PyTorch 陣列的**自動降維特性**與**廣播機制的右對齊原則**有關。
* **不加 `keepdim=True**`：PyTorch 橫向加總後會把該維度消除，原本 `(27, 27)` 的二維矩陣會降維成沒有方向性的一維橫排 `(27,)`。當進行除法時，廣播機制會右對齊，導致矩陣「躺下來」誤變成**每一直欄**除以該數值，機率表徹底壞掉，出現前幾行加總 `P.sum(1)[:5]` 不等於 `1.0` 而是 `tensor([7.0225, 4.4142...])` 的數值災難。
* **加 `keepdim=True**`：將形狀鎖定在 `(27, 1)` 的直條向量。廣播機制會橫向複製，正確地讓**每一橫行除以各自的總和**，此時 `P.sum(1)` 會呈現完美的 `tensor([1.0000, 1.0000...])`。
* *註：若將除號 `/=` 誤寫成 `=`（如 `P = P.sum(1, keepdim=True)`），會直接把整個次數矩陣整台丟掉覆蓋，導致 `P` 退化成 `(27, 1)` 的單欄向量，進到迴圈後會因為寬度不足而百分之百永遠只抽到 `0`，造成迴圈在第一步瞬間中斷。*



### ❓ 提問 6：什麼是 One-hot encodings？為什麼不能直接把整數餵給神經網路？

* **原理解析**：神經網路底層是連續的矩陣乘法運算（$X \times W$），只懂程度大小，不懂分類代號。如果直接輸入整數代號（如 `a=1, b=2, c=3`），神經網路會誤以為 `c` (3) 的價值是 `a` (1) 的 3 倍大，且 `c` 比 `b` 更接近 `a`。
* **解法**：One-hot 編碼把數字轉成長度為 27 的稀疏向量（如 `c` 代表 `[0, 0, 0, 1, 0...0]`）。與權重矩陣相乘時，任何數乘以 0 都是 0，它能像「類別開關」一樣，精準且獨立地啟動對應字母的權重，抹除任何不存在的大小順序暗示。

### ❓ 提問 7：`torch.tensor` 和 `torch.Tensor` 的差別是什麼？

* **原理解析**：
* **`torch.tensor`（小寫 t）**：工廠函數（Factory Function）。會自動推斷資料型態（餵整數變 `int64`，餵浮點數變 `float32`），並且永遠執行深拷貝（Deep Copy），在 ML 開發與 Debug 時最為安全、推薦使用。
* **`torch.Tensor`（大寫 T）**：PyTorch 的 Tensor 類別建構子（Class Constructor），本質是 `torch.FloatTensor` 的別名。預設會把所有資料強制轉成 `float32`（這會使索引整數被強轉成浮點數而導致陣列報錯）。此外，若傳入兩個整數如 `torch.Tensor(3, 4)`，它代表的是建立一個 $3 \times 4$ 未初始化的隨機殘留記憶體矩陣。



### ❓ 提問 8：解構矩陣乘法中，權重 `W` 設為 `(27, 1)` 與 `(27, 27)` 的差異？為什麼 `(27, 1)` 不行？

* **原理解析**：輸入的 One-hot 向量 `xenc` 形狀為 `(1, 27)`。
* **當 `W` 是 `(27, 1)**`：矩陣乘法 $(1, 27) \times (27, 1) \rightarrow (1, 1)$。這是一個「只有 1 個神經元」的網路，無論輸入什麼字母，最終只能吐出 1 個孤零零的數字答案。你無法用 1 個數字去把機率同時分配給下一個可能的 27 個字母。它因為能力太弱，根本無法表達一張完整的機率表。
* **當 `W` 是 `(27, 27)**`：矩陣乘法 $(1, 27) \times (27, 27) \rightarrow (1, 27)$。這是一個「擁有 27 個神經元並行計算」的神經網路層。利用 `@` 的乘法特性，它會精準抽取出 `W` 矩陣中的某一橫行（27個權重分數），這 27 個數字剛好可以拿來代表預測下一個字是誰的原始分數（Logits），完美繼承並擁有了原本 $27 \times 27$ 機率表的所有表達能力。
